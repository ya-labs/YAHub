# Back-end da V1

## Objetivo

Este documento registra a especificação técnica inicial do back-end da V1 do YA Hub.

O objetivo do back-end é centralizar o acesso aos dados do portal, entregar contratos limpos para o front-end, persistir os dados editoriais do YA Hub e preparar a base para autenticação administrativa, cadastro de projetos, cadastro de membros e integrações futuras.

## Decisão de Stack

O back-end da V1 será implementado em .NET para priorizar qualidade, previsibilidade e velocidade de entrega no projeto principal da YA LABS.

Stack inicial:

- .NET 9.
- ASP.NET Core Web API.
- C#.
- Arquitetura orientada a DDD.
- PostgreSQL.
- Entity Framework Core 9.
- Npgsql Entity Framework Core Provider.
- JWT para autenticação administrativa.
- Docker para banco local de desenvolvimento.
- Azure como destino provável de hospedagem futura.

## Estrutura

O back-end fica em:

```text
Back-end/
|-- YaHub.Backend.sln
`-- src/
    |-- YaHub.Api/
    |-- YaHub.Application/
    |-- YaHub.Domain/
    `-- YaHub.Infrastructure/
```

Responsabilidades dos projetos:

- `YaHub.Api`: entrada HTTP, endpoints, autenticação, configuração de pipeline e composição de dependências.
- `YaHub.Application`: casos de uso, contratos de aplicação, DTOs, validações de fluxo e `Result<T>`.
- `YaHub.Domain`: entidades, value objects, regras de domínio, enums e contratos essenciais do domínio.
- `YaHub.Infrastructure`: persistência, integrações externas, clientes da GitHub API, serviços técnicos e implementações de contratos.

## Fluxo de Dependências

```text
YaHub.Api
  -> YaHub.Application
  -> YaHub.Infrastructure

YaHub.Infrastructure
  -> YaHub.Application
  -> YaHub.Domain

YaHub.Application
  -> YaHub.Domain

YaHub.Domain
  -> sem dependências internas
```

O domínio não deve depender de ASP.NET Core, Entity Framework, banco, GitHub API ou infraestrutura.

## Padrão de Resposta

A API deve usar um envelope simples inspirado no padrão `Result<T>`:

```json
{
  "result": true,
  "message": null,
  "data": {}
}
```

Falhas de regra de negócio devem retornar:

```json
{
  "result": false,
  "message": "Mensagem de erro.",
  "data": null
}
```

Erros inesperados devem ser tratados por middleware ou filtros globais quando essa camada for implementada.

## Banco de Dados

O banco da aplicação será PostgreSQL.

Durante o desenvolvimento local, o banco deve rodar em PostgreSQL local, preferencialmente via Docker. Em ambiente hospedado, a previsão é usar Azure.

A persistência inicial está configurada na camada `YaHub.Infrastructure`, usando:

- `YaHubDbContext`;
- `DbSet<User>`;
- `DbSet<Project>`;
- `DbSet<Member>`;
- migrations em `YaHub.Infrastructure/Persistence/Migrations`;
- registro do contexto na API com `UseNpgsql`.

O arquivo `appsettings.Development.json` deve ser usado apenas localmente e não deve ser versionado, pois pode conter connection string, usuário e senha do banco local.

Exemplo de configuração local:

```json
{
  "ConnectionStrings": {
    "DefaultConnection": "Host=localhost;Port=5432;Database=yahub;Username=yahub;Password=yahub"
  }
}
```

Credenciais, tokens e connection strings reais não devem ser versionados. Em ambiente local, a configuração pode vir de `appsettings.Development.json` ou user secrets. Em ambientes hospedados, deve vir de variáveis de ambiente ou configuração segura da plataforma.

Comandos principais de persistência:

```powershell
dotnet ef migrations add NomeDaMigration `
  --project .\src\YaHub.Infrastructure `
  --startup-project .\src\YaHub.Api `
  --output-dir Persistence\Migrations

dotnet ef database update `
  --project .\src\YaHub.Infrastructure `
  --startup-project .\src\YaHub.Api
```

## Autenticação

A autenticação administrativa deve usar:

- login com email e senha;
- hash de senha;
- JWT;
- proteção dos endpoints administrativos;
- cadastro administrativo controlado antes de produção.

Endpoints iniciais previstos:

```text
POST /api/login
POST /api/register
```

## Projetos

O cadastro de projetos deve permitir que o admin mantenha dados editoriais do YA Hub e associe cada projeto a um repositório GitHub.

Endpoints previstos:

```text
GET /api/projects
GET /api/projects/{slug}
GET /api/admin/github/repositories
POST /api/admin/github/repositories/resolve
GET /api/admin/projects
POST /api/admin/projects
PUT /api/admin/projects/{id}
DELETE /api/admin/projects/{id}
```

Os endpoints administrativos devem exigir autenticação.

## Membros

Membros também devem ser cadastrados pelo painel admin na V1.

Endpoints previstos:

```text
GET /api/members
GET /api/members/{slug}
GET /api/admin/members
POST /api/admin/members
PUT /api/admin/members/{id}
DELETE /api/admin/members/{id}
```

## Integração com GitHub

Na V1, o back-end deve consultar a GitHub API para obter dados técnicos e públicos dos repositórios.

Dados esperados:

- nome do repositório;
- proprietário;
- descrição;
- linguagem principal;
- URL pública;
- datas relevantes;
- atividade recente simples, quando fizer sentido.

Dados editoriais, como destaque, categoria, status, visibilidade e descrição tratada, devem vir do banco do YA Hub.

## Limites da V1

Não fazem parte do escopo imediato:

- painel administrativo completo e sofisticado;
- upload de imagens;
- workflow de aprovação;
- permissões administrativas granulares;
- criação automática de Pull Requests;
- renderização de documentação interna dos projetos;
- integração real com Discord;
- integração real com Spotifolio;
- leitura de `.yahub/project.json`;
- deploy final na Azure.
