# Back-end da V1

## Objetivo

Este documento registra a especificação técnica inicial do back-end da V1 do YA Hub.

O objetivo do back-end é centralizar o acesso aos dados do portal, entregar respostas tratadas para o front-end e preparar a base para autenticação, cadastro de projetos e integrações futuras.

Esta documentação descreve a direção técnica atual. Ela deve ser atualizada conforme a implementação amadurecer.

## Stack

Stack inicial do back-end:

- Java 21.
- Spring Boot.
- Spring Web MVC.
- Spring Data JPA.
- Spring Security.
- PostgreSQL.
- Lombok.
- Maven.
- Docker para banco local de desenvolvimento.
- Azure Database for PostgreSQL como destino previsto para ambiente hospedado.

## Responsabilidades

O back-end da V1 deve:

- expor endpoints HTTP para o front-end;
- padronizar respostas da API;
- persistir dados administrativos no PostgreSQL;
- autenticar usuários administrativos;
- proteger senhas com hash seguro;
- preparar a base para cadastro e manutenção de projetos;
- isolar o front-end de detalhes internos do banco e de integrações externas;
- manter uma estrutura simples, legível e evolutiva.

## Arquitetura Inicial

O fluxo principal segue o modelo em camadas:

```text
Controller
  -> Service
    -> Repository
      -> PostgreSQL
```

Responsabilidades por camada:

- `controller`: recebe requisições HTTP, lê parâmetros/body e devolve `ResponseEntity`.
- `service`: concentra regras de negócio e orquestra operações.
- `repository`: acessa o banco usando Spring Data JPA.
- `model`: representa entidades persistidas.
- `dtos`: representa contratos de entrada e saída da API.
- `mapper`: converte DTOs em entidades e entidades em DTOs.
- `config`: registra configurações do Spring, como segurança e password encoder.
- `shared`: guarda estruturas compartilhadas, como `Result<T>`.

## Estrutura de Pacotes

Estrutura inicial prevista:

```text
com.yalabs.yahub
|-- config
|-- controller
|-- dtos
|   |-- requests
|   `-- responses
|-- mapper
|-- model
|-- repository
|-- service
`-- shared
```

Novos pacotes devem ser adicionados apenas quando houver necessidade real.

## Padrão de Resposta

A API deve responder usando o envelope `Result<T>`:

```json
{
  "result": true,
  "message": null,
  "data": {}
}
```

Falhas de regra de negócio devem retornar:

```json
{
  "result": false,
  "message": "Mensagem de erro.",
  "data": null
}
```

O `Result<T>` deve ser usado principalmente na camada de service e retornado pelos controllers.

Erros inesperados de infraestrutura ou bugs não devem ser escondidos com `catch` genérico em todos os métodos. A evolução esperada é criar um handler global de exceções quando a API tiver mais superfície.

## Banco de Dados

O banco da aplicação será PostgreSQL.

No desenvolvimento local, o banco pode rodar em Docker:

```text
localhost:5433 -> container PostgreSQL:5432
```

Em ambiente hospedado, a previsão é usar Azure Database for PostgreSQL.

Configuração local esperada:

```properties
spring.datasource.url=jdbc:postgresql://localhost:5433/yahub
spring.datasource.username=yahub
spring.datasource.password=yahub
```

Durante a fase inicial de aprendizado e modelagem, pode ser usado:

```properties
spring.jpa.hibernate.ddl-auto=update
```

Antes de estabilizar ambientes compartilhados ou produção, a decisão deve ser revista para uso de migrations com Flyway ou Liquibase.

## Entidades Iniciais

### User

Entidade inicial para autenticação administrativa.

Campos previstos:

- `id`: identificador UUID.
- `name`: nome do usuário.
- `email`: email único.
- `passwordHash`: hash da senha.
- `createdAt`: data de criação.

Senha nunca deve ser salva em texto puro.

## Autenticação

A autenticação inicial deve usar:

- login com email e senha;
- cadastro de usuário administrativo;
- hash de senha com BCrypt;
- proteção de endpoints administrativos com JWT.

Fluxo de login:

```text
POST /api/login
  -> buscar usuário por email
  -> comparar senha enviada com passwordHash usando BCrypt
  -> retornar token JWT e dados públicos do usuário

POST /api/register
  -> validar dados de cadastro
  -> gerar passwordHash usando BCrypt
  -> persistir usuário administrativo
  -> retornar dados públicos do usuário criado
```

Mensagens de erro de login devem evitar revelar se o email existe ou se apenas a senha está incorreta.

O cadastro administrativo pode entrar na V1 para acompanhar a base inicial do back-end. Antes de produção, ele deve ser protegido ou limitado para evitar criação pública aberta de administradores.

## Endpoints Iniciais

Endpoints já considerados na base inicial:

```text
GET /health
POST /api/login
POST /api/register
GET /api/users
DELETE /api/users/{id}
```

Endpoints administrativos de usuário devem ser revisados antes de ambiente hospedado para garantir autenticação, permissão e estratégia segura de cadastro.

## Cadastro de Projetos

O objetivo funcional antecipado para a V1 é permitir cadastro de projetos pelo back-end.

Endpoints previstos:

```text
GET /api/projects
GET /api/projects/{slug}
GET /api/admin/projects
POST /api/admin/projects
PUT /api/admin/projects/{id}
DELETE /api/admin/projects/{id}
```

Os endpoints administrativos de projetos devem ficar protegidos por autenticação JWT.

Dados iniciais previstos para projeto:

- identificador UUID;
- slug;
- nome de exibição;
- categoria: `produto` ou `ecossistema`;
- descrição curta;
- descrição completa;
- status;
- visibilidade: `publico` ou `oculto`;
- linguagem principal;
- URL do repositório;
- URL do site, quando existir;
- URL da documentação, quando existir;
- indicação de destaque;
- ordem de exibição;
- data de criação;
- data de atualização.

Status iniciais:

- `ideia`;
- `planejamento`;
- `desenvolvimento`;
- `ativo`;
- `pausado`;
- `arquivado`.

## Cadastro de Membros

Membros também devem ser cadastrados pelo painel admin na V1.

O modelo final ainda pode ser refinado durante a implementação, mas o back-end deve prever persistência própria para os membros exibidos no portal.

Endpoints previstos:

```text
GET /api/members
GET /api/members/{slug}
GET /api/admin/members
POST /api/admin/members
PUT /api/admin/members/{id}
DELETE /api/admin/members/{id}
```

Os endpoints administrativos de membros devem ficar protegidos por autenticação JWT. As rotas públicas devem retornar somente membros visíveis.

## Documentação dos Projetos

A documentação interna dos projetos não entra na V1.

Na V1, quando houver necessidade de consulta, o YA Hub deve apontar para o GitHub do repositório. O back-end pode guardar links de documentação, mas não precisa importar nem renderizar documentação do projeto.

O produto deve preparar integração futura com o YABook, como uma ação `Abrir com o YABook`, quando esse suporte existir.

## Segurança

Regras iniciais:

- não salvar senha em texto puro;
- não retornar `passwordHash` em DTOs de resposta;
- não registrar senha, hash ou token em log;
- liberar apenas rotas públicas necessárias no `SecurityFilterChain`;
- proteger endpoints administrativos quando JWT for adicionado;
- usar variáveis de ambiente ou configuração segura para credenciais fora do desenvolvimento local.

## Logs

Services podem usar SLF4J via Lombok:

```java
@Slf4j
```

Logs devem registrar eventos técnicos relevantes, sem expor dados sensíveis.

Exemplos aceitáveis:

- usuário cadastrado com id;
- tentativa de cadastro com email já existente;
- falha de regra de negócio sem senha ou token.

## Decisões Atuais

- O back-end da V1 será implementado em Java/Spring Boot.
- O banco será PostgreSQL.
- O desenvolvimento local pode usar PostgreSQL em Docker.
- A hospedagem futura deve considerar Azure Database for PostgreSQL.
- A primeira estratégia de persistência usa Spring Data JPA.
- O hash de senha deve usar BCrypt.
- O padrão `Result<T>` será usado para respostas da API.
- JWT será usado para autenticação administrativa.
- Login e cadastro administrativo entram na V1.
- Os metadados oficiais de projetos e membros ficarão no banco do YA Hub.
