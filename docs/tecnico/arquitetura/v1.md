# Arquitetura técnica da V1

## Objetivo

A V1 do YA Hub será um produto full-stack para apresentar a YA LABS, seus projetos, membros e atividade recente, incluindo painel administrativo básico para manter projetos e membros.

O objetivo técnico desta fase é criar uma base simples, clara e preparada para evolução, com front-end e back-end no mesmo repositório.

As decisões técnicas reais da V1 devem ficar neste documento ou em documentos técnicos específicos do YA Hub. Padrões reutilizáveis de documentação, processo e design system devem permanecer no YABook.

## Estrutura do repositório

A estrutura prevista para a V1 é:

```text
YA-HUB/
|-- apps/
|   `-- api/
`-- docs/
```

O monorepo facilita a coordenação entre front-end, back-end, admin e documentação técnica enquanto o produto ainda está em fase inicial.

## Stack prevista

### Front-end

- Angular.
- TypeScript.

O admin da V1 deve ficar na mesma aplicação front-end do YA Hub.

### Back-end

O back-end da V1 será implementado com Java e Spring Boot, adaptado ao domínio do YA Hub:

- Java 21.
- Spring Boot.
- Spring Web MVC.
- Spring Data JPA.
- PostgreSQL.
- Lombok.
- Maven.
- Docker para banco local de desenvolvimento.
- Azure Database for PostgreSQL como destino previsto para ambiente hospedado.

## Estrutura conceitual do back-end

```text
apps/api/src/main/java/com/yalabs/yahub/
|-- config/
|-- controller/
|-- dtos/
|-- mapper/
|-- model/
|-- repository/
|-- service/
`-- shared/
```

Essa estrutura deve servir como base inicial. Novas pastas só devem ser adicionadas quando houver necessidade real.

## Fluxo de dados

```text
Front-end
  -> Back-end Spring Boot
      -> GitHub API
      -> PostgreSQL
      -> dados editoriais cadastrados no admin
      -> tratamento dos dados
      -> resposta padronizada para o front-end
```

O front-end não deve chamar a GitHub API diretamente na V1. Ele deve consumir apenas os contratos definidos pelo back-end.

## Responsabilidades do front-end

- Construir a Home institucional e o portal público do YA Hub.
- Exibir dashboard do portal, projetos, membros e atividade recente.
- Separar projetos entre Produtos e Ecossistema.
- Construir o admin básico dentro da mesma aplicação.
- Consumir os endpoints do back-end.
- Tratar estados de carregamento, erro e vazio.
- Aplicar a direção visual inicial do YA Hub.
- Manter componentes desacoplados da origem real dos dados.

## Responsabilidades do back-end

- Consultar a GitHub API.
- Persistir projetos e membros no PostgreSQL.
- Autenticar administradores.
- Cadastrar usuários administrativos de forma controlada.
- Expor endpoints administrativos para projetos e membros.
- Centralizar regras de montagem dos dados públicos.
- Padronizar respostas para o front-end.
- Proteger tokens quando necessário.
- Aplicar cache quando fizer sentido.
- Preparar base para YABook, Spotifolio e Discord em versões futuras.

## Integração com GitHub

Na V1, o GitHub será usado para obter dados técnicos e públicos dos repositórios da organização.

Exemplos de dados esperados:

- repositórios;
- descrições;
- linguagens principais;
- links;
- datas de atualização;
- commits ou atividade recente simples.

Dados que o GitHub não cobre bem, como destaque, status editorial e relação com membros, devem vir do banco do YA Hub.

## Documentação dos projetos

A documentação interna dos projetos não faz parte da V1.

Na V1, o portal deve apontar para a documentação disponível no GitHub do repositório. A integração futura com o YABook deve ser preparada conceitualmente, por exemplo com uma ação `Abrir com o YABook` quando esse suporte existir.

## Cache

O back-end pode usar cache para reduzir chamadas repetidas à GitHub API.

O cache não precisa ser sofisticado na V1. A intenção é evitar excesso de chamadas externas e criar uma base para evolução futura.

## Deploy e infraestrutura

O back-end deve nascer preparado para deploy em container com Docker.

A hospedagem futura pode ser feita na Azure, com apoio de plataforma/infraestrutura quando essa etapa for planejada.

## Fora da V1

Não fazem parte desta fase:

- documentação interna dos projetos;
- upload de imagens;
- criação automática de Pull Requests;
- permissões administrativas granulares;
- workflow de aprovação;
- integração real com Spotifolio;
- bot ou automações no Discord;
- consumo de `.yahub/project.json`.

## Decisão técnica principal

A V1 deve priorizar uma base full-stack funcional e compreensível, sem antecipar complexidade das versões futuras.

Se uma decisão técnica for aceita durante a implementação e tiver impacto duradouro, ela deve ser registrada em documentação própria ou em ADR futura, seguindo a orientação do YABook.
